# Phase 8.7 — 客户端实施方案：Job/Subagent 子流可观测性

> Status: 方案已评审通过，待实施。对应后端 PRD：[p8.7-job-observability.md](p8.7-job-observability.md)
> （重点 §8 客户端契约、§9 交付清单）。协议基线：[client_integration_v1.md](client_integration_v1.md)。
> 交互参考：Claude Code 的 subagent UX —— 子任务在对话 turn card 内显示为一张卡片，
> macOS 点击展开为**右侧面板**（呈现为一个完整的 turn card，带活跃状态），iOS 点击展开为 **sheet**。

## 0. 现状核对（起点比 §9 清单假设的要好）

对照代码逐项核实过：

| 后端 PRD §9 清单项 | 客户端现状 |
|---|---|
| 未知 `kind` 不能崩 | **已满足**。`AgentEvent.from(wire:)`（[AgentEvent.swift:206](../Sources/AgentKit/Core/AgentEvent.swift)）对未知 kind 返回 `nil` 直接丢弃，`WireFrame` 全字段可选。零改动就不会崩 |
| `task_started`/`task_finished` 处理 | **管线已全通**：解码（[AgentEvent.swift:177](../Sources/AgentKit/Core/AgentEvent.swift)）→ reducer 建 `SubagentExecPayload` 节点（[ExecutionReducer.swift:600](../Sources/AgentKit/Core/RuntimeEngine/ExecutionReducer.swift)）→ 但投影层只降级成一行系统文本（[TimelineProjection.swift:244](../Sources/AgentKit/Core/RuntimeEngine/TimelineProjection.swift)"Sub-agent: prompt → result"），**没有可展开卡片** |
| `job_wait` 工具卡 | 走通用工具卡状态机（按 `call_id` 聚合），预期零改动，只需验证 |
| 轮询取数 | [RuntimeHTTPClient.getEvents](../Sources/AgentKit/Core/RuntimeHTTPClient.swift) 已存在，但**不支持 `since=` 参数**，且现有事件 DTO 里没有 seq 字段（见 §4 问题 1） |
| 侧边栏基建 | Inspector 体系现成：[InspectorSelection](../Sources/AgentKit/Features/Inspector/InspectorSelection.swift) 枚举 + `InspectorView` 分发，加一个 case 即可挂新面板 |

关键架构事实：事件流是单向管线
`WireFrame → AgentEvent → RuntimeEngine(actor) → ExecutionReducer → ExecutionGraph → TimelineProjection → RuntimeSnapshot → SwiftUI`，
另有一条 legacy 镜像路径 `ConversationState.reduce`（[ConversationState.swift:370](../Sources/AgentKit/Features/Conversation/Models/ConversationState.swift)
也处理 task 事件）——**所有新增分支要在两条路径都加**，否则新旧 UI 不一致。

## 1. 工作项拆解

### WI-1：协议层三个新 kind（对应后端 Phase A，约 0.5–1 天）

- `AgentEvent` 新增三个 case：`jobStarted(turnID:jobID:command:)`、`jobOutput(jobID:chunk:)`、
  `jobFinished(jobID:status:exitCode:text:)`，`from(wire:)` 里加分支。**`WireFrame` 大概率零改动**：
  `sessionId`、`chunk`、`err` 字段都已存在（这是决策点 1 投"复用 `session_id`"的直接理由，见 §3）。
- `ExecutionReducer`：把现有 `SubagentExecPayload` 泛化为 `ChildStreamPayload`（加一个
  `kind: .task | .job` 判别），`job_started`/`job_finished` 与 `task_started`/`task_finished`
  走同一对 handler——后端 PRD §8.0 说的"一套渲染逻辑、两处收益"落到代码上就是这一步。
  `job_output` 若被转进父流则忽略（决策点 2 说明只转 bracket，不转 output；防御性处理以防万一）。
- `ConversationState.reduce`（legacy 路径）加同样的分支。
- 测试：`ExecutionReducerTests` 加 job 生命周期用例（started→finished 就地更新同一节点、
  乱序/缺 started 时不崩），等后端冻结线格式后在 `docs/protocols/fixtures` 补 golden 解码用例。

### WI-2：`job_wait` 工具卡（对应后端 Phase B，约 0.5 天，纯验证 + 可选糖）

- 验证通用卡渲染正确（running 态长时间悬挂是这张卡的常态，确认无超时类 UI 误判）。
- 可选糖：在 `ToolCard`/`ToolTranscriptPresenter` 里对 `tool_name == "job_wait"` 给专属文案
  （"正在等待后台任务完成…" + 计时器，`ThinkingTimerView` 可复用）。不阻塞任何 phase。

### WI-3：子流查看器（对应后端 Phase C 客户端侧，主力工作量，约 3–5 天，**可立即开工**）

分四块：

1. **传输抽象**（这是"后端就绪后只换传输层"的落点）：

   ```swift
   protocol ChildStreamTransport: Sendable {
       func events(since: Int) async throws -> [WireFrame]   // 轮询实现：现在就能写
       // WS attach 实现：Phase C 后端就绪后新增，协议不变
   }
   ```

   第一版 `PollingChildStreamTransport`：1s 间隔轮询 `getEvents(id, since:)`，收到
   `job_finished`/`task_finished` 后停止，前台激活时才轮询（iOS 后台挂起）。
   另做一个 fixture 回放实现（读 JSON 数组按时间间隔吐帧），viewer 全链路可以先在
   mock 下开发和截图验收，**不依赖后端**。

2. **`ChildStreamViewModel`**：每个子流一个实例，内部复用一个独立的 `RuntimeEngine`——
   task 子流的事件词汇表和主会话完全相同（tool_started/thinking/…），reducer 原样可用；
   job 子流只有三个 kind，同样能被 WI-1 的分支消化。**不需要新 reducer**。

3. **查看器视图**（交互对齐 Claude Code 的 subagent UX，见文首参考）：
   - 用子流自己的 `RuntimeSnapshot.timeline` 喂现有的 timeline/工具卡渲染器
     （`NativeTranscriptView` / `ConversationTimelineView` 的节点视图）——渲染层全复用。
   - **macOS**：`InspectorSelection` 加 `case childStream(sessionID: String, kind: ChildStreamKind)`，
     挂进 `InspectorView` 的 switch，在**右侧面板**展开。面板呈现为一个**完整的 turn card**：
     顶部是子任务标题（task 的委派 prompt / job 的 command）+ 返回按钮，主体是子流自己的
     step-by-step 事件时间线（工具卡、thinking、输出分块实时追加），**带活跃状态指示**
     （running 时顶部显示 spinner/计时器，`RuntimeSnapshot.turnStartedAt` 机制可复用）。
   - **iOS**：点击入口卡以 **sheet** 呈现同一个查看器视图（`presentationDetents` 给
     medium/large 两档），内容与 macOS 面板完全同源，只是容器不同。

4. **父时间线入口卡**：把 `TimelineProjection` 里 subagent 的"系统文本行"升级为真正的
   可展开卡片节点（新 `ExecutionNodeKind`），**内嵌在所属 turn card 中**（对齐 Claude Code：
   主对话只有一张折叠卡片）。卡片显示：图标 + 子任务一句话摘要（prompt/command 截断）、
   running 时的活跃态（spinner + 已运行时长）、结束后的结果摘要行（task 的结论 /
   job 的 exit 状态），点击触发上面 3 的展开。job 与 task 共用这张卡。

## 3. 对后端 PRD §8.4 三个决策点的客户端投票

1. **子流 id 字段：投"复用 `session_id`"**。`WireFrame.sessionId` 已存在，reducer 对
   task/job 统一取子流 id，零 DTO 改动、零分支判断。DTO 命名别扭的问题客户端内部用
   `childStreamID` 语义化命名消化掉，不需要线格式配合。
2. **bracket 事件转发进父流：强烈投"转发"**。不转发的话客户端只能从 `run_command` 的
   observation 文本里抠 job id（脆弱且没有结束通知），入口卡就做不了。事件密度不是问题：
   每个 job 只多 2 条事件（不含 `job_output`），reducer 是逐事件 O(1) 就地更新，且
   `RuntimeEngine` 已有 16ms 合帧。
3. **Phase C 端点形态：两者都能接受，微倾向"独立轻量端点"**，附加条件：**独立端点吐的
   事件帧格式必须与 `/v1/conversations/{id}/events` 完全一致**（同一个 `WireFrame` 解码器
   直接用）。transport 抽象本来就允许 task 走 `/v1/conversations/{id}/stream`、job 走
   `/v1/jobs/{id}/stream` 两种 URL。若走复用 `conversation.Manager` 方案，请确保 job 会话
   **不出现**在 `GET /v1/conversations` 列表里（否则会话侧栏会混入 job，客户端要加过滤逻辑）。

## 4. 需要后端额外确认的两个问题（§8.4 之外，调研客户端代码时发现）

1. **`since=N` 的 N 从哪来？** 现有事件线格式没有 `seq` 字段（v1 明确说 seq 是 v2 预留），
   客户端轮询时无法从已收到的事件里得知"下一次该从哪续"。需要后端确认：是
   "N = 已收到的事件条数"（要求 seq 从 1 连续无空洞），还是在事件上补一个可选 `seq` 字段？
   **建议后者**——顺手也解决了 client_integration_v1.md §5.8 断线重连不能精确续传的老问题。
2. **`job_finished` 携带什么终态字段？** 后端 Sink 接口草案里有 `status`/`exitCode`，
   请在冻结 golden 前给一版字段清单，客户端好定 `ChildStreamPayload` 的形状；特别是失败态
   （非零 exit / 被 kill）在入口卡和查看器顶栏要区分显示。
